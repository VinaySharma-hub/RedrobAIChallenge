## 2. Pipeline overview

1. **Load** — `load_candidates()` reads `candidates.jsonl` (or `.gz`) line
   by line into a list of dicts matching `candidate_schema.json`.
2. **Flatten to text** — `candidate_to_text()` concatenates profile,
   career-history (title/company/industry/description per role), skills
   (name + proficiency), education, and certifications into one lowercased
   string per candidate, used both for TF-IDF and for keyword counting.
3. **TF-IDF retrieval score** — a `TfidfVectorizer` (unigrams+bigrams,
   English stopwords, `min_df=2`, max 50K features) is fit across the JD
   text plus all candidate texts; `tfidf_score` is cosine similarity
   between each candidate vector and the JD vector.
4. **Rule-based component scores:**
   - `technical_rule_score` — weighted blend of retrieval-term, ML-term,
     and production-term hit counts (0.45 / 0.30 / 0.25), with an extra
     penalty if the title contains "computer vision" and retrieval
     evidence is weak.
   - `experience_score` — banded score peaking at 1.0 for 5–9 years
     (the JD's stated band), tapering outward.
   - `location_score` — 1.0 for JD-preferred cities, 0.75 for
     India + willing-to-relocate, 0.43 for India only, 0.13 otherwise.
   - `behavioral_score` — composite of profile completeness, open-to-work
     flag, recruiter response rate, response time, GitHub activity,
     recruiter saves, interview completion rate, offer acceptance rate,
     verification flags, and notice-period bands (bonus ≤30/60d, neutral
     60–90d, penalty 90–120d, larger penalty ≥120d).
5. **Penalty functions** (subtracted from the composite):
   - `trap_penalty` — expert-proficiency-with-zero-duration skills,
     too many expert skills, career-duration vs. stated-experience
     mismatch, malformed education date ranges.
   - `consulting_only_penalty` — 0.25 if every career-history company
     matches a known consulting firm, 0.12 if ≥75% do.
   - `keyword_stuffing_penalty` — penalizes many AI-related skill tags
     with little or no corresponding evidence in career-history
     descriptions.
   - `inactivity_penalty` — penalizes large gaps between the latest
     career-history `end_year` and the current year.
   - `title_penalty` — flat penalties for "junior"/"intern"/"trainee" in
     the current title.
   - `advanced_honeypot_penalty` — the most thorough check:
     graduation-year vs. stated-experience consistency, skill-duration vs.
     total-experience consistency, expert-skill-count thresholds,
     seniority/experience mismatches (e.g. "Staff"/"Principal" title with
     <5y experience), career-duration vs. stated-experience consistency,
     overlapping employment date ranges, and future-dated career entries.
6. **Final composite:**
   ```
   final_score = 0.30 * tfidf_score
               + 0.30 * technical_rule_score
               + 0.15 * behavioral_score
               + 0.10 * experience_score
               + 0.05 * location_score
               - (trap_penalty + consulting_only_penalty
                  + keyword_stuffing_penalty + inactivity_penalty
                  + title_penalty + advanced_honeypot_penalty)
   ```
   clamped at a floor of 0.0.
7. **Reasoning generation** — `generate_reasoning()` builds a sentence from
   years/title plus up to 4 matched retrieval/ML terms found in the
   candidate's flattened text, followed by a behavioral-signals sentence
   (open-to-work, response rate, notice period, GitHub score), truncated to
   450 characters.
8. **Rank and write** — sorted by `final_score` descending, ties broken by
   `candidate_id` ascending; top 100 written to CSV with sequential ranks
   1–100.

## 3. Design rationale

- **TF-IDF instead of embeddings** for the retrieval component, to stay
  within the no-GPU / CPU-only / 5-minute compute budget on the full
  100K-candidate file without needing to manage local embedding-model
  weights or warm-up cost.
- **Additive penalties rather than hard exclusion** for trap/honeypot
  signals, so a candidate with one borderline flag is demoted rather than
  removed outright — softer than a hard filter, at the cost of needing
  penalty weights to be well-calibrated (see limitations).
- **`advanced_honeypot_penalty` checks structural consistency** (graduation
  year, career-duration sums, overlapping employment, future dates) rather
  than just keyword patterns, aimed at the dataset's stated honeypot
  construction method ("subtly impossible profiles").

## 4. Known limitations

- **JD is hand-typed, not parsed from `job_description.docx`.** `JD_TEXT`
  is a manually written paraphrase of the JD's key terms. The TF-IDF
  retrieval signal reflects what we judged to be the JD's important terms,
  not an automated extraction — be upfront about this if asked how JD
  understanding is "automated" in this approach.
- **Duplicate `inactivity_penalty` definition** — cells 21 and 23 in the
  notebook define the same function name identically; the second silently
  shadows the first. Functionally harmless since they're identical, but
  worth cleaning up before any further iteration.
- **Inconsistent "current date" reference.** `inactivity_penalty` uses
  `datetime.now().year` (real wall-clock time), while
  `advanced_honeypot_penalty`'s overlap/future-date checks use a hardcoded
  `datetime(2026, 6, 11)`. This makes `inactivity_penalty` output
  non-deterministic depending on when the script runs — pin both to the
  same reference date before relying on this for reproducible scoring.
- **Reasoning text is templated.** `generate_reasoning()` builds from a
  fixed sentence skeleton; Stage 4 manual review penalizes templated
  reasoning.
- **Penalty weights are hand-set, not tuned** against any held-out signal —
  they reflect judgment calls about relative severity, not calibration.
- **This notebook does NOT produce the competition submission.** The
  `team_redrob_submission.csv` in the repo root is produced by the
  agents pipeline in `Agents_work/`.

## 1. Architecture

The pipeline runs candidates through a sequence of independent agent
functions in `agents_dev.py` (and optionally `candidate_ranking_agents.py`),
each producing structured output that downstream agents or the final scorer
in `ranker.py` consume:

| Agent | Responsibility |
|---|---|
| `jd_understanding_agent(jd_text)` | Parses the JD into must-have / good-to-have keyword groups, strong/weak evidence phrases, and structured disqualifier flags (consulting-only, CV/speech/robotics-only, pure-research-only, title-chasers, no-visa-sponsorship) parsed directly from the JD's own "explicitly do NOT want" language. |
| `candidate_audit_agent(candidate, jd_analysis)` | Matches a candidate's career/skills/summary/title text against the JD analysis; flags keyword-stuffing, weak/course-like evidence, junior titles, and oversized skill lists; emits a promote/demote/keep recommendation. |
| `trap_detection_agent(candidate)` | Honeypot detection — flags candidates whose `profile.years_of_experience` doesn't reconcile with `sum(career_history[*].duration_months)`. Also flags JD-explicit disqualifiers: consulting-only career, CV/speech/robotics-only with no NLP/IR exposure, pure-research-no-production, and title-chasing (rapid Senior→Staff→Principal hops). |
| `behavioral_signal_agent(candidate, today=None)` | Converts `redrob_signals` + location/relocation into a single availability/hireability modifier: open-to-work flag, inactivity (`last_active_date`), recruiter response rate, notice-period bands matching the JD's stated preference (≤30d preferred, buyout up to 30d, 60–90d marginal, >90d penalized), and country/relocation/visa-sponsorship fit. |
| `feature_engineering_agent(audits, trap_results)` | Aggregates per-candidate audit/trap output across the batch into a list of suggested scoring rules (e.g. exclude on honeypot, exclude on consulting-only, demote on pure-research). This is a meta-agent for batch inspection, not a per-candidate scorer. |
| `reasoning_agent(candidate, score_data, audit, trap_result, behavior_result)` | Builds the human-readable `reasoning` column text — title/experience summary, career-evidence sentence, title-fit sentence, behavioral-signal sentence, then every audit risk flag, trap flag, and behavior flag as `Concern:` / `Note:` clauses, truncated to 1200 chars. |

`ranker.py` owns:
- loading `candidates.jsonl`,
- building each candidate's `career_text` / `skills_text` / `summary_text` / `title_text` fields,
- calling the agents in sequence per candidate,
- combining agent outputs into a single `final_score`,
- sorting and writing the top 100 to `team_redrob_submission.csv`.

### Candidate dict shape expected by the agents

```python
{
    "id": "CAND_XXXXXXX",
    "career_text": "<lowercased concatenation of career_history titles+descriptions>",
    "skills_text": "<lowercased concatenation of skill names>",
    "summary_text": "<lowercased profile.summary>",
    "title_text": "<lowercased profile.current_title>",
    "raw": <full candidate JSON, conforming to candidate_schema.json>,
}
```

## 2. Design rationale

- **Rules over a black-box model.** Given the no-GPU / no-network / 5-minute
  CPU budget, and given that several of the JD's most decisive
  disqualifiers (consulting-only career, CV/robotics-without-NLP, pure
  research without production deployment) are precise enough to express as
  explicit logic, they are encoded directly rather than relying on an
  embedding model to infer them from career text.
- **Honeypot detection is structural, not lexical.** `trap_detection_agent`
  reconciles `profile.years_of_experience` against the sum of
  `career_history[*].duration_months` — the dataset's stated honeypot
  construction method per the README bundle ("subtly impossible profiles").
- **Behavioral signals are a modifier, not a primary score.** A
  "perfect-on-paper but unreachable" candidate is penalized but not
  excluded outright — `behavioral_signal_agent` returns a 0–1 score
  intended to adjust, not gate, the base relevance score.
- **Reasoning is generated from the same structured agent outputs used for
  scoring** — every `Concern:`/`Note:` clause in the CSV reasoning column
  traces back to a specific flag a reviewer can verify against the
  candidate's raw profile.

## 4. Compute constraints

Per `submission_spec.md` Section 3: no GPU, no network calls during the
ranking step, ≤5 minutes, ≤16GB RAM, CPU only. All agent logic is pure
Python (string search, arithmetic, date parsing) with no external API
calls or model inference — runtime on the full 100K file should be
benchmarked before submission (see Known Issues).
